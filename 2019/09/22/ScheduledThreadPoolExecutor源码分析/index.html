类结构


如图：ScheduledThreadPoolExecutor类

  • 继承ThreadPoolExecutor，所以该类具备线程池的所有功能，也可以用来执行普通任务。
  • 实现ScheduledExecutorService接口，ScheduledExecutorService接口扩展了ExecutorService接口，添加了延时任务和定时任务相关的接口，下面我们看一下ScheduledExecutorService接口。

如上，ScheduledThreadPoolExecutor在普通线程池的基础上增加了执行延时任务和定时任务的功能。下面看一下它的源码，通过源码来研究一下它的执行流程。

构造方法

如上，ScheduledThreadPoolExecutor实际上创建了一个任务队列为DelayedWorkQueue队列的线程池，这个队列是无界队列，所以线程池的最大线程数和空闲存活时间等参数都是无效的。

内部类

ScheduledFutureTask

这个类继承FutureTask，是线程池要执行的任务主体。

DelayedWorkQueue

这个类严格来说不属于内部类，而属于嵌套类，是一个堆结构的阻塞队列，代码就不详细看了，感兴趣的可以自行研究一下。这里暂时只需要知道它是一个基于堆结构的阻塞队列就可以了。

主要方法

public ScheduledFuture<?> schedule(Runnable command, long delay, TimeUnit unit)

这里将任务加入到阻塞队列后，线程池中的worker线程就会从阻塞队列里取任务来执行，下面我们看一下取任务的逻辑，这块逻辑在ThreadPoolExecutor类中

这里调用getTask方法来获取任务，获取不到阻塞等待，下面我们看一下getTask方法。

可以看到，需要从工作队列里拿取任务，对于ScheduledThreadPoolExecutor来说，工作队列就是DelayedWorkQueue，下面让我们看一下该阻塞队列的poll和take方法。

如上，DelayedWorkQueue是一个堆结构，堆顶是最早可以执行的任务，所以只需要从堆顶获取元素，判断元素在当前时间是否可以执行，不能执行则等待。
take方法和poll方法逻辑一样，只是去掉了超时等待超时时间的判断。
下面看一下ScheduledFutureTask的run方法。

如果不是周期任务，执行结束，如果是周期任务，把任务重新放入阻塞队列中，等待下一次执行。

public ScheduledFuture schedule(Callable callable, long delay, TimeUnit unit)
与上面的方法相同，只是换成了Callable版本。

public ScheduledFuture<?> scheduleAtFixedRate(Runnable command, long initialDelay, long period, TimeUnit unit)

与上面的重载方法差不多，只多了一句将当前任务保存到outerTask变量中的逻辑。

public ScheduledFuture<?> scheduleWithFixedDelay(Runnable command, long initialDelay, long delay, TimeUnit unit)
与上面的逻辑相同。

总结

ScheduledThreadPoolExecutor执行流程如下：

  • 1.将任务包装成ScheduledFutureTask，然后添加到阻塞队列里。
  • 2.线程从阻塞队列中获取任务，任务执行时间没有到就阻塞等待。
  • 3.如果任务是周期任务，就在执行结束后，重新添加该任务到阻塞队列中。
