JDK里的线程池中的不足

jdk里提供了一个线程池的实现ThreadPoolExecutor，它的执行流程如下：

我们可以看到：

  1. 当现有线程数小于核心线程数的时候，创建核心线程执行任务。
  2. 当现有线程数大于核心线程数的时候，会直接将任务放入阻塞队列里。
  3. 当现有线程数大于核心线程数，并且阻塞队列已满的情况下，会继续创建线程执行任务。
  4. 当现有线程数达到最大线程数的时候，执行拒绝策略，有多种策略可以选择，默认直接抛异常。

上面的步骤1是没有问题的，但是步骤2和步骤3对于大部分的io密集型场景不是很合适。不合适的原因主要有以下2点：

  1. 核心线程数默认是不会空闲超时回收的，设置的太大会造成线程池在很空闲的时候也会占用大量资源。
  2. 如果当前线程数已经大于核心线程数的话，把任务放入到阻塞队列里，只有当阻塞队列满了之后才会继续创建新线程来执行，而如果我们希望创建更多线程来提升系统的并行度，就需要将阻塞队列设置的很小，而设置的太小又容易丢弃大量任务。

基于上面的原因我们可以发现，JDK原生线程池并不是很适合io密集型的场景，对应io密集型的场景，我们往往希望当当前线程数大于核心线程数的时候优先创建线程，当达到最大线程数的时候再进入阻塞队列里排队等待，也就是希望上面的流程2和流程3调换位置。

tomcat里的线程池优化

tomcat作为主流的web容器和servlet容器，面对的场景是有大量网络io的io密集型业务，所以也很自然的会遇到上面的问题，那tomcat是怎么样来解决这个问题的呢？
下面就让我们看一下tomcat对JDK原有线程池的扩展和改写。

扩展JDK的线程池

新增成员变量submittedCount,用来表示提交到线程池里还没有执行完成的任务数量

重写核心方法execute

Tomcat 线程池的 execute 方法会调用 Java 原生线程池的 execute 去执行任务，如果总线程数达到 maximumPoolSize，Java 原生线程池的 execute 方法会抛出 RejectedExecutionException 异常，但是这个异常会被 Tomcat 线程池的 execute 方法捕获到，并继续尝试把这个任务放到任务队列中去；如果任务队列也满了，再执行拒绝策略。

定制版任务队列

如上，当当前线程数小于最大线程数是，会先判断需要被执行的任务数是否小于线程数，如果是的话，说明当前线程池有空闲线程，那么我们就直接加入的任务队列里，让空闲线程来取任务执行；否则，我们就创建新的线程来执行任务，而不是像JDK里的线程池一样，把它加入任务队列里。

总结

tomcat线程池通过实现定制化的任务队列和重写核心execute方法，修改了JDK中线程池的默认执行流程，具体为：

  1. 当当前线程数大于核心线程数，并且当前线程池没有空闲线程的情况下，会优先创建线程来执行任务，而不是加入到任务队列里。
  2. 当达到最大线程数时，没有直接执行拒绝策略处理，而且再次尝试加入到任务队列里，加入失败再执行拒绝策略处理。

通过以上的对JDK原生线程池的扩展和重写，tomcat实现了一个更加适合io密集型任务的线程池。
